# MTProxy Auto-Installer

Установка и защита MTProxy (telemt) с маскировкой под реальный HTTPS.  
Трафик неотличим от обычного HTTPS — DPI и сканеры ТСПУ получают настоящий сертификат домена-маски.

> Протестировано на **Debian 13 (trixie)**

---

## Скрипты и порядок запуска

Репозиторий содержит 4 скрипта. Запускать в указанном порядке:

```
┌─────────────────────────────────────────────────────────┐
│                                                         │
│  ШАГ 1 — На VPS-сервере                                 │
│  bash mtproxy_install.sh                                │
│  Устанавливает telemt, генерирует ключи, мониторинг     │
│                                                         │
│  ШАГ 2 — Настройка домена (рекомендуется)               │
│  Купить домен → A-запись → сервер → LE сертификат       │
│  Обязательно для мобильных операторов (Билайн и др.)    │
│  Подробнее: DOMAIN_SETUP.md                             │
│                                                         │
│  ШАГ 3 — На VPS-сервере                                 │
│  bash tspu_server_check.sh                              │
│  Проверяет что прокси виден из России и не детектируем  │
│                                                         │
│  ШАГ 4 — На VPS-сервере                                 │
│  bash server_harden.sh                                  │
│  Защищает сервер: SSH, fail2ban, автообновления         │
│                                                         │
│  ШАГ 5 — На машине клиента (в России)                   │
│  bash tspu_client_check.sh <ip> <port> <secret>         │
│  Диагностирует подключение через ТСПУ                   │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

| Скрипт | Где запускать | Когда |
|---|---|---|
| `mtproxy_install.sh` | VPS | Первый запуск, при смене IP |
| `tspu_server_check.sh` | VPS | После установки и при подозрении на блокировку |
| `server_harden.sh` | VPS | После проверки прокси, один раз |
| `tspu_client_check.sh` | Клиент (Россия) | При проблемах с подключением |

Подробная документация: **[TSPU_Readme.md](TSPU_Readme.md)**, **[HARDEN_Readme.md](HARDEN_Readme.md)**, **[DOMAIN_SETUP.md](DOMAIN_SETUP.md)**

## Быстрый старт

### 1. Установить прокси

```bash
bash mtproxy_install.sh
# или с параметрами:
bash mtproxy_install.sh 443 browser.yandex.com you@example.com
```

Получить ссылки для подключения:
```bash
curl -s http://127.0.0.1:9091/v1/users | jq
```

Результат — ссылка вида:
```
tg://proxy?server=1.2.3.4&port=443&secret=ee47baa7...
```

> ⚠️ telemt инициализирует соединения с Telegram ~90 секунд. Ссылки и порт 443 появятся после полного старта.

---

### 2. Настроить свой домен (рекомендуется для всех провайдеров)

**Почему домен критически важен:** DPI мобильных операторов (Билайн, МТС) проверяет что SNI в TLS ClientHello соответствует реальной A-записи DNS. Без своего домена — `Telegram handshake timeout` в мобильных сетях.

```
Клиент Telegram
    │ TLS SNI = proxy.example.com
    ▼
103.35.x.x  ← DNS: proxy.example.com → 103.35.x.x  ✓ совпадает
    │
    ├─ Без секрета → Let's Encrypt сертификат (легитимный)
    └─ С секретом  → telemt → Telegram DC
```

Подробная инструкция: **[DOMAIN_SETUP.md](DOMAIN_SETUP.md)**

Краткий порядок:
```bash
# 1. Добавить A-запись: proxy.example.com → IP сервера
# 2. Открыть порт 80
ufw allow 80/tcp

# 3. Выпустить сертификат
apt-get install -y certbot nginx
certbot certonly --standalone -d proxy.example.com \
    --non-interactive --agree-tos -m you@example.com

# 4. Настроить nginx заглушку и telemt конфиг
# (подробнее в DOMAIN_SETUP.md)
```

---

### 3. Проверить сервер из России

```bash
bash tspu_server_check.sh
```

Ключевые результаты:
```
[2] RESULT: NOT BLOCKED — port 443 accessible from Russia ✓
[5] ✓ Real TLS — certificate verified OK (TCP Splice working)
[6] ✓ Max latency drift: +0.0ms — stable
```

Если всё зелёное — прокси работает и не детектируется.

---

### 4. Защитить сервер

> ⚠️ Сначала настройте SSH-ключи — скрипт отключит парольный вход.

```bash
bash server_harden.sh 8443   # укажите ваш SSH-порт
```

Проверить в новом терминале что SSH работает, затем:
```bash
ufw delete allow 22/tcp   # закрыть старый порт
```

---

### 5. Диагностика с клиента (при проблемах)

Запустить на машине в России:

```bash
bash tspu_client_check.sh 1.2.3.4 443 ee47baa7...secret...
```

## Почему не официальный Docker-образ Telegram

Официальный `telegrammessenger/proxy` (2020 года) **детектируется ТСПУ** с апреля 2026:

| Проблема | Следствие |
|---|---|
| Уникальный JA3/JA4 fingerprint — порядок cipher suites не совпадает ни с одним браузером | DPI идентифицирует MTProxy за секунды |
| Без секрета возвращает ошибку или обрывает соединение | Сканер ТСПУ фиксирует аномалию |

Наблюдаемый паттерн блокировки: **40ms → 600ms → полный блок за 5–15 часов.**

**telemt** решает это через TCP Splice:

```
Клиент с секретом   →  telemt  →  Telegram MTProxy
Клиент без секрета  →  telemt  →  реальный сайт-маска
ТСПУ сканер         →  получает реальный TLS-сертификат
```

| Режим | DPI fingerprint | Время до блокировки |
|---|---|---|
| Plain MTProto | Сигнатура в первых пакетах | Секунды |
| FakeTLS `ee` (официальный Docker) | Уникальный JA3/JA4 | Минуты |
| telemt + чужой домен | Реальный TLS, DNS не совпадает | Часы |
| **telemt + свой домен + LE** | **Реальный TLS, DNS совпадает** | **Дни / не блокируется** |

## Как ТСПУ блокирует прокси

```
Новое подключение IP:port
        │
        ▼
Stage 1 (0–15 мин): DPI анализирует сессию
  Распознан → лог → SPFS → ЦСУ
  protocols capacity 2-10%: дроп пакетов
  40ms → 600ms (симптом: задержка растёт)
        │
        ▼
Stage 2 (5–15 мин после Stage 1):
  IP:port → блок-лист → полная блокировка
```

**Что блокируется:** IP:port — не секрет, не содержимое.  
**Смена секрета не помогает.**

При блокировке — действовать в этом порядке:
1. Сменить IP сервера (наиболее эффективно)
2. Сменить домен: `bash mtproxy_install.sh 443 другой-домен.com`
3. Сменить порт: `bash mtproxy_install.sh 8443`

## Ограничения мобильных сетей

MTProxy работает иначе в мобильных сетях — это важно понимать до настройки.

### Белый список операторов

Ряд российских мобильных операторов проверяет соответствие SNI в TLS ClientHello и реальной A-записи DNS. Если домен в секрете не резолвится в IP сервера — хэндшейк обрывается.

Наблюдаемое поведение без своего домена:
- TCP-соединение с сервером **устанавливается** (телефон доходит до прокси)
- MTProto-хэндшейк **не завершается** — оператор обрывает его
- В логах telemt: `Connection closed with error: Telegram handshake timeout`
- По WiFi того же провайдера — прокси работает нормально

### Операторы и совместимость

| Оператор | Без своего домена | Со своим доменом |
|---|---|---|
| Билайн (Vimpelcom, AS16345) | ❌ Handshake timeout | ✅ Работает |
| МТС | ⚠️ Зависит от региона | ✅ Работает |
| Мегафон | ⚠️ Зависит от региона | ✅ Работает |
| Ростелеком | ✅ Работает | ✅ Работает |
| Tele2 | ⚠️ Зависит от региона | ✅ Работает |

### Диагностика: мобильная сеть или сервер?

```bash
journalctl -u telemt -f | grep -E "WARN|ERROR|handshake"
```

| Что видно в логах | Диагноз | Решение |
|---|---|---|
| Нет строк с мобильным IP | IP заблокирован полностью | Сменить хостер |
| `Telegram handshake timeout` | SNI не совпадает с DNS | Настроить свой домен |
| Нет ошибок | Проблема на клиенте | Проверить ссылку |

### Узнать мобильный IP и оператора

На телефоне → браузер → `https://ifconfig.me` → скопировать IP, затем:

```bash
curl -s "http://ip-api.com/json/ВАШ_МОБИЛЬНЫЙ_IP?fields=country,isp,as"
```

### Архитектурное ограничение

Telegram **не поддерживает звонки через MTProxy** — только через SOCKS5. Это техническое ограничение протокола.

## Мониторинг прокси

Устанавливается автоматически при `mtproxy_install.sh`.  
Запускается каждые 15 минут, проверяет задержку с российских узлов.

```
< 150 ms   → OK
> 150 ms   → WARN_LATENCY (лог)
> 400 ms   → WARNING + email (Stage 1 ТСПУ)
недоступен → CRITICAL + email (Stage 2, полный блок)
```

```bash
# Запустить вручную
mtproxy-monitor

# Лог
tail -f /var/log/mtproxy-monitor.log

# Настроить email
echo 'you@example.com' > /etc/telemt/notify-email
```

## Управление прокси

```bash
# Статус
systemctl status telemt

# Логи
journalctl -u telemt -f

# Все ссылки для подключения
curl -s http://127.0.0.1:9091/v1/users | jq

# Добавить пользователя (без перезапуска)
openssl rand -hex 16   # сгенерировать секрет
# добавить в /etc/telemt/telemt.toml → [access.users]

# Перезапуск
systemctl restart telemt
```
