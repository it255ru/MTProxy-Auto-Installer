# TSPU Checkers

Два диагностических скрипта для проверки MTProxy/VPN в условиях **ТСПУ (АСБИ)** — российской системы фильтрации трафика.

---

## Скрипты

| Скрипт | Где запускать | Назначение |
|---|---|---|
| `tspu_server_check.sh` | На VPS-сервере | Проверяет видимость сервера из России, работу прокси, детектируемость |
| `tspu_client_check.sh` | На машине клиента | Диагностирует подключение через ТСПУ к вашему серверу |

---

## tspu_server_check.sh

### Запуск

```bash
# Стандартный (тест порта 443)
bash tspu_server_check.sh

# Другой порт
bash tspu_server_check.sh 8443
```

### Что проверяет

| Шаг | Проверка | Что означает |
|---|---|---|
| 1 | Geo & IP-классификация | `hosting=false` — IP не помечен как датацентр, меньше шансов на автоматическую блокировку |
| 2 | Порт из России (check-host.net) | Реальная проверка с российских узлов — самый важный тест |
| 3 | Локальные открытые порты | Какие сервисы слушают |
| 4 | Определение прокси-процесса | telemt / xray / sing-box / официальный MTProxy |
| 5 | TCP Splice верификация | Сканер без секрета должен получить **реальный** сертификат домена-маски |
| 6 | Тест деградации (25 сек) | Измеряет drift задержки — ранний признак ТСПУ Stage 1 |
| 7 | DNS разрешение | Скорость резолва с сервера |
| 8 | Качество сети | BBR, jitter, ошибки интерфейса, память |
| 9 | IP-репутация | AS в блок-листах, классификация IP |

### Ключевые результаты

**Шаг 5 — TCP Splice (критично для telemt):**
```
✓ Real TLS — certificate verified OK (TCP Splice working)
  CN (issuer): GlobalSign
  Verify return code: 0 (ok)
✓ DPI scanner sees real www.bing.com — JA3/JA4 = browser fingerprint
```
Это означает: DPI-сканер, подключившийся без секрета, получает настоящий сертификат домена-маски.
`CN: GlobalSign` — это промежуточный CA для `*.bing.com`, всё корректно.
`Verify return code: 0 (ok)` — формат OpenSSL 3.x (Debian 12/13), эквивалентен `SSL certificate verify ok`.

**Шаг 6 — тест деградации:**
```
Baseline latency : 42ms
Holding connection for 25 sec...
  t+ 5.0s  latency=45ms  drift=+3ms
  t+10.0s  latency=48ms  drift=+6ms
  t+15.0s  latency=312ms drift=+270ms  ← HIGH (ТСПУ degradation?)
✗ Max drift: +270ms — ТСПУ packet dropping (Stage 1)
```
Рост drift выше +200ms — Stage 1 ТСПУ активен, полная блокировка может последовать через 5–15 минут.

## tspu_client_check.sh

### Запуск

```bash
# MTProxy с секретом
bash tspu_client_check.sh 103.35.189.80 443 ee47baa7...secret...

# Без секрета (VLESS / общая диагностика)
bash tspu_client_check.sh 103.35.189.80 443
```

### Что проверяет

| Шаг | Проверка | Что означает |
|---|---|---|
| 1 | Расположение клиента | ISP, AS — важно для понимания через какой ТСПУ идёт трафик |
| 2 | Базовый TCP-коннект | Доступен ли порт вообще |
| 3 | Тест деградации (25 сек) | Измеряет drift задержки и ищет 19-секундный разрыв |
| 4 | TLS handshake / TCP Splice | Получает ли клиент реальный сертификат домена-маски |
| 5 | Формат MTProxy-секрета | Декодирует тип и домен из секрета |
| 6 | DNS на стороне клиента | Блокировка DNS — частая проблема за ТСПУ |
| 7 | Traceroute | Где именно теряются пакеты |
| 8 | Latency & jitter | 10 замеров — определяет стабильность соединения |
| 9 | Итог | Сводка и маппинг симптомов |

### Диагностика по симптомам

| Симптом пользователя | Что смотреть |
|---|---|
| Telegram не подключается | Шаг 2 — TCP timeout |
| Подключается, потом отваливается через ~20 сек | Шаг 3 — drop_time 14-24с |
| Медленно, с задержкой | Шаг 3 — drift, Шаг 8 — jitter |
| `wsarecv: forcibly closed by remote host` | Шаг 3 — ТСПУ 19s signature |
| `dns: exchange failed` / `EOF` | Шаг 6 — DNS заблокирован |
| Видео заикается | Шаг 8 — jitter > 50ms |
| `vision: not a valid TLS connection` | Шаг 4 — сервер не поддерживает Vision |
| Секрет не принимается | Шаг 5 — формат секрета |

### Интерпретация теста деградации (Шаг 3)

```
Baseline latency : 85ms
Holding connection for 25 sec...
  t+ 5.0s  92ms   drift=+7ms
  t+10.0s  94ms   drift=+9ms
  t+15.0s  98ms   drift=+13ms
✓ Connection held 25s — no forced drop
✓ Max drift: +13ms — latency stable
```
→ ТСПУ не детектирует протокол. Соединение стабильно.

```
Baseline latency : 90ms
  t+ 5.0s  95ms   drift=+5ms
  t+10.0s  388ms  drift=+298ms  ← HIGH (ТСПУ Stage 1?)
  t+14.2s  Connection dropped
✗ DROPPED at 14.2s — ТСПУ 19-second DPI signature!
✗ Max drift: +298ms — ТСПУ packet dropping (Stage 1)
```
→ DPI распознал протокол, применил деградацию и разорвал соединение.

## Как работает ТСПУ (для понимания результатов)

```
Подключение нового IP:port
        │
        ▼
Stage 1 (0–15 мин): DPI-движок анализирует сессию
        │  Размеры пакетов, частота, паттерны
        │  Многофакторный анализ, статистика
        │
        ├─ Не распознан → трафик идёт свободно ✓
        │
        └─ Распознан → лог → SPFS → ЦСУ
                │
                │ protocols capacity: 2–10%
                │ Дроп пакетов → TCP retransmit
                │ Задержка растёт (40ms → 600ms)
                │
                ▼
Stage 2 (5–15 мин): ЦСУ формирует «очищенный список»
        │  IP:port → в блок-лист фильтра (behavior: block)
        │  IP:port → Eco Highway BGP
        │
        └─ Полная блокировка
```

**С telemt TCP Splice:**
- DPI видит TLS-соединение с реальным сертификатом `www.bing.com`
- JA3/JA4 fingerprint = настоящий браузерный
- Stage 1 распознавание значительно затруднено
- Блокировка по IP:port (Stage 2) всё ещё возможна при накоплении статистики

## Примечание: `proxy=true` в ip-api

Поле `proxy: true` в базе ip-api.com означает, что через этот IP **ранее** проходил proxy/VPN трафик
(возможно, предыдущий арендатор VPS). Это **не источник блокировок для ТСПУ**.

ТСПУ не использует ip-api.com. Критически важны только два поля:
- `hosting=false` — IP не классифицирован как датацентровый
- AS не в блок-листах РКН

Если `proxy=true` вас беспокоит — проверьте детально: https://www.ipqualityscore.com/ip-reputation/YOUR_IP

## Сравнение режимов

| Режим | DPI fingerprint | Stage 1 | Stage 2 |
|---|---|---|---|
| Plain MTProto (без префикса) | Сигнатура в первых пакетах | Секунды | Минуты |
| DD-обфускация | Статистический паттерн | Минуты | 5–15 мин |
| FakeTLS `ee` (официальный Docker) | Уникальный JA3/JA4 | Минуты | 5–15 мин |
| **telemt TCP Splice** | **Реальный браузерный TLS** | **Часы/дни** | **Дни** |
